DDIA笔记-02数据模型与查询语言

发表于 2022-11-10 更新于 2024-02-13 分类于 数据密集型应用系统设计读书笔记 Valine：

本节对比了各种不同的数据模型，即不同类型的数据库是如何组织数据，表达数据间的关系的。具体的数据模型有：

  • 关系型，关系型数据库的数据模型。关系（表）是元组（行） 的集合，支持集合上的交集，并集，笛卡尔积等操作。通过Join操作可以反映数据之间简单的一对多、多对一和多对多关系。但是Join操作成本较高，表达一对多关系不够直观，也表达不了复杂的多对一和多对多关系；数据分散在多个表中，局部性不够好；强schema，增加字段成本很高；
  • 文档型：天然的一对多、树形嵌套关系。但是支持Join的能力较弱，对多对一和多对多关系的支持有限。具有良好的访问局部性；弱 schema，支持动态增加字段；
  • 图模型：为了更好支持多对多关系，提出了图模型。

另外，还介绍了两种查询语言：

  • 声明式：描述控制逻辑而非执行流程。比如SQL，CSS等；
  • 命令式：描述命令的执行过程，用一系列语句来不断改变状态。比如常用的编程语言C/C++等；

A data model is an abstract model that organizes elements of data and standardizes how they relate to one another and to the properties of real-world entities.

https://en.wikipedia.org/wiki/Data_model

数据模型：如何组织数据，如何标准化数据间的关系，如何关联现实。

大多数应用程序是通过一层一层叠加数据模型来构建的。每一层都面临的关键问题是：如何将其用下一层来表示？例如：

  1. 应用开发者，通过对象或数据结构以及作用于其上的API对现实世界进行建模。
  2. 当存储这些数据结构时，可以采用通用数据模型（例如JSON或XML文档、关系数据库中的表或图模型）来表示。
  3. 数据库系统开发者需将上述数据模型组织为内存、硬盘或网络中的字节流，并提供多种操作数据的方法。
  4. 硬件工程师需要将字节流表示为二极管的电位（内存）、磁场中的磁极（磁盘）、光纤中的光信号（网络）。

基本思想就是，每层都通过提供一个简洁的数据模型来隐藏下层的复杂性。本章将介绍用于数据存储和查询的通用数据模型（上面第2点）。将比较关系模型、文档模型和基于图的数据模型。还将讨论多种查询语言并比较它们的使用场景。

关系模型与文档模型

现在最著名的数据模型可能是SQL，它基于1970年提出的关系模型，数据被组织成关系，在SQL中称为表，其中每个关系是元组（表中的行）的无序集合。当时的其他数据库迫使应用开发人员考虑数据的内部表示。关系模型的目标就是将实现细节隐藏在更简洁的接口后面。

NoSQL的诞生

多年来，在数据存储和查询方面存在着其他许多竞争技术。进入21世纪， NoSQL（最初表示 Non-SQL，后来有人转解为 Not only SQL）成为推翻关系模式主导地位的又一个竞争者。

这样几个因素驱动了NoSQL数据库：

  1. 处理更大数据集：更强伸缩性、更高吞吐量
  2. 开源免费的兴起：冲击了原来把握在厂商的标准
  3. 特化的查询操作：关系数据库难以支持的，比如图中的多跳分析
  4. 表达能力更强：关系模型约束太严，限制太多

对象-关系不匹配

现在多数应用都采用面向对象的编程语言，数据存储在关系表中，代码中的对象与表、行和列的数据库模型间需要一个转换层。虽然有对象-关系映射（ORM）框架减少了转换层的样板代码量，但他们并不能完全隐藏两个模型间的差异。

例如，图2-1展示了如何在关系模式中表示简历。多数人有一个以上的工作，并且可能有多个教育阶段和任意数量的联系信息。用户与这些项目之间存在一对多的关系，关系模型很难直观的表示一对多的关系。

像简历这样的数据结构，它主要是一个自包含（对外部无依赖）的文档（document），因此用JSON表示非常合适，JSON比XML更简单。面向文档的数据库（如MongoDB）都支持该数据模型。

JSON表示比图2-1的多表模式有更好的局部性。关系模式中读取简历，要么执行多个查询（通过user_id查询每个表），要么在users表及其从属表间执行混乱的多路联结。而JSON表示法，所有的信息都在一个地方，一次查询就够了。

在简历的例子中，文档模型还有几个优势：

  1. 模式灵活：可以动态增删字段，如工作经历。
  2. 结构表达语义：简历与教育经历、职业信息等隐含一对多的树状关系可以被JSON的树状结构明确表达出来。

多对一与多对多的关系

简历的例子中，region_id和industry_id定义为ID，而不是如”Greater Seattle Area” 和“Philanthropy”这样的纯文本字符串形式。相比于在用户界面让用户自己输入文本字段，让用户从下拉列表或自动填充器中进行选择会更有优势：

  1. 统一样式：所有用到相同概念的地方都有相同的拼写和样式
  2. 避免歧义：可能有同名地区
  3. 易于修改：如果一个地区改名了，我们不用去逐一修改所有引用他的地方
  4. 本地化支持：如果翻译成其他语言，可以只翻译名字表。
  5. 更好搜索：列表可以关联地区，进行树形组织

使用ID的好处是，因为它对人类没有任何直接意义，所以永远不需要直接改变。任何对人类有意义的东西都可能在将来发生变更。当直接存储文本时，使用它的每条记录中都保存了一份信息。修改时所有的副本也都需要更新。这会导致更多写入开销，并且存在数据不一致的风险。消除这种重复正是数据库规范化的核心思想。

文档模型不适合表达多对一的关系（许多人生活在同一地区，许多人在同一行业工作）。对于关系数据库，由于支持联结操作，可以很方便地通过ID来引用其他表中的行，而文档数据库中，一对多的树状结构无需联结，支持联结通常也很弱。

如果数据库本身不支持联结，则必须在应用程序代码中，通过对数据库进行多次查询来模拟联结。

文档数据库是否在重演历史？

层次模型

20 世纪 70 年代，IBM 的信息管理系统IMS采用了相当简单的数据模型，称为层次模型。它与文档数据库使用的JSON模型有一些显著的相似之处。层次模型可以很好地支持一对多关系，但它支持多对多关系则有些困难，而且不支持联结。

为了解决层次模型的局限，人们提出了各种解决方案，最突出的是：关系模型和网络模型。

网络模型

在层次模型的树结构中，每个记录只有一个父结点；而在网络模型中，一个记录可能有多个父结点。多对一和多对多都可以由路径来表示。访问记录的唯一方式是顺着元素和链接组成的链路进行访问，这个链路叫访问路径。难度犹如在n维空间中进行导航。它们使查询和更新数据库变得异常复杂而没有灵活性。

关系模型

关系模型定义了所有数据的格式：关系（表）只是元组（行） 的集合。没有复杂的嵌套结构，也没有复杂的访问路径。可以读取表中的任何一行或者所有行，支持任意条件查询。可以指定某些列作为键并匹配这些列来读取特定行。可以在任何表中插入新行，而不必担心与其他表之间的外键。

在关系数据库中， 查询优化器自动决定以何种顺序执行查询，以及使用哪些索引。这些选择实际上等价于“访问路径”， 但最大的区别在于它们是由查询优化器自动生成的，而不是由应用开发人员所维护。

关系模型的一个核心要点是：只需构建一次查询优化器，然后使用该数据库的所有应用程序都可以从中受益。相比网络模型，关系模型的查询语句和执行路径相解耦。

文档数据库的比较

文档数据库是某种方式的层次模型：在其父记录中保存了嵌套记录（一对多关系，如上面JSON中的positions、education和contact_info），而不是存储在单独的表中。

在表示多对一和多对多的关系时，关系数据库和文档数据库并没有根本的不同：两种情况下，相关项都由唯一的标识符引用，该标识符在关系模型中被称为外键，在文档模型中被称为文档引用。

关系数据库与文档数据库对比（本节只关注数据模型中的差异）

支持文档数据模型的主要论点是模式灵活性，由于局部性而带来较好的性能，对于某些应用来说，它更接近于应用程序所使用的数据结构。关系模型则强在联结操作、多对一和多对多关系更简洁的表达上。

文档模型中的模式灵活性

大多数文档数据库，以及关系数据库中的JSON支持，都不会对文档中的数据强制执行任何模式。没有模式意味着可以将任意的键值添加到文档中。

文档数据库有时被称为无模式，这具有误导性，更准确的术语应该是读时模式（数据的结构是隐式的，只在读取时才解释），与之对应的是写时模式（关系数据库的一种传统方法，模式是显式的，并且数据库确保数据写入时必须遵循）。

读时模式类似编程语言中的动态（运行时） 类型检查，而写时模式类似于静态（编译时）类型检查。

当应用需改变数据格式时，读时模式和写时模式间的差异变得明显。例如，当前用户的全名存储在一个字段中，现在想分别存储名字和姓氏。在文档数据库中，只需使用新字段来编写新文档，井在应用层来处理读取旧文档的情况。例如：

查询的数据局部性

文档通常编码为JSON 、XML或其二进制变体（如MongoDB的BSON）的连续字符串进行存储。如果应用需频繁访问整个文档，则存储局部性有性能优势。如果数据被分在多个表中，则需进行多次索引查找来检索所有数据，中间可能需要更多的磁盘IO井花费更多的时间。

局部性优势仅适用需访问文档大部分内容的场景。如果应用只访问其中的一小部分，则对于大型文档数据来讲就有些浪费。对文档进行更新时，通常会重写整个文档，只有修改量不改变文档大小时， 原地覆盖更新才更有效。因此，通常建议文档应该尽量小且避免写入时增加文档大小。这些性能方面的不利因素大大限制了文档数据库的适用场景。

文挡数据库与关系数据库的融合

MySQL和PostgreSQL开始支持JSON。随着时间的推移，似乎关系数据库与文档数据库变得越来越相近，融合关系模型与文档模型是未来数据库发展的一条很好的途径。

图状数据模型

如果数据大多是一对多关系（树结构数据）或者记录之间没有关系，文档模型是最合适的。如果是多对多的关系，关系模型能够处理简单的多对多关系，但是随着数据之间的关联越来越复杂，将数据建模转化为图模型会更加自然。

图由两种对象组成：顶点（也称为结点或实体）和边（也称为关系或弧）。很多数据可以建模为图。典型的例子包括：

  • 社交网络：顶点是人，边指示哪些人彼此认识。
  • Web 图：顶点是网页，边表示与其他页面的HTML链接。
  • 公路或铁路网：顶点是交叉路口，边表示他们之间的公路或铁路线。

有很多著名的算法可以在这些图上运行。例如，汽车导航系统搜索道路网中任意两点之间的最短路径。

图也提供了单个数据存储区中保存完全不同类型对象的一致性方式。

为方便理解，可以将图存储看作由两个关系表组成，一个用于顶点，另一个用于边，为每个边存储头部和尾部顶点。如果想要顶点的入边或出边的集合，可以分别通过head_vertex或tail_vertex来查询edges表。

关于图模型一些值得注意的地方：

  1. 任何顶点都可以连接到其他任何顶点。没有模式限制哪种事物可以或不可以关联。
  2. 给定某个顶点，可以高效地得到它的所有入边和出边，从而遍历图，即沿着这些顶点链条一直向前或向后（这就是为什么示例2-2中在tail_vertex和head_vertex列上都建立索引的原因）。
  3. 通过对不同类型的关系使用不同的标签，可以在单个图中存储多种不同类型的信息，同时仍然保持整洁的数据模型。

相对于关系型数据来说，可以在同一个图中保存异构类型的数据和关系，给了图极大的表达能力！这种表达能力，根据图中的例子，包括：

  1. 对同样的概念，可以用不同结构表示。如不同国家的行政划分。
  2. 对同样的概念，可以用不同粒度表示。比如 Lucy 的现居住地和诞生地。
  3. 可以很自然的进行演化。例如，可以把这个图扩展到包括许多关于Lucy和Alain的其他信息。可以用它来表示他们的任何食物过敏（通过为每个过敏源引入顶点，以及人与过敏源之间的边来表示过敏），并将过敏源与顶点的集合联结。

将异构的数据容纳在一张图中，可以通过图遍历，轻松完成关系型数据库中需要多次 Join 的操作。

SQL中的图查询

前面看到可以用 SQL 存储点和边，以表示图。那表示也可以用 SQL 进行图查询。虽然存在一些困难，在关系数据库中，通常会预先知道查询中需要哪些join操作。而对于图查询，在找到要查找的顶点之前，可能需要遍历数量未知的边，也就是说， join操作数量并不是预先确定的。

对于示例2-4 ，这主要集中在Cypher查询的[:WITHIN*0..]规则中。一个人的LIVES_IN边可以指向任何类型的位置，例如街道、城市、地区、区域、国家等。城市可以位于（ WITHIN） 某地区内，地区可以位于（WITHIN）某州内等。LIVES_IN边可以直接指向正在查找的位置顶点， 也可以是位置层次结构中删除的某些层。

Cypher可以用[:WITHIN*0..]非常简洁地表达这个情况：它表示“沿着一个WITHIN边，遍历零次或多次”，就像正则表达式中的＊运算符（ 表示匹配零次或多次）那样。

SQL:l999标准以后， 查询过程中这种可变的遍历路径可以使用称为递归公用表表达式（即WITH RECURSIVE） 来表示。目前PostgreSQL 、IBM DB2, Oracle和SQL Server等都支持该技术，但与Cypher丰目比，语法仍显得非常笨拙。

三元存储与SPARQL

三元存储模式几乎等同于属性图模型，在三元存储中，所有信息都以非常简单的三部分形式存储（主体，谓语，客体） 。例如， 在三元组（吉姆， 喜欢， 香蕉）中，吉姆是主体， 喜欢是谓语（动词），香蕉是客体。

三元组的主体相当于图中的顶点。而客体则是以下两种之一：

  1. 原始数据类型中的值，如字符串或数字。在这种情况下， 三元组的谓语和客体分别相当于主体（顶点） 属性中的键和值。例如， (Lucy, αge,33)就好比是顶点lucy，具有属性｛”age”: 33｝ 。
  2. 图中的另一个顶点。此时，谓语是图中的边，主体是尾部顶点， 而客体是头部顶点。例如， 在(Lucy, marriedTo, alain)中，主体Lucy和客体alain都是顶点，井且谓语marriedTo是连接二者的边的标签。
